# @nextcalc/database

Shared Prisma 7 database package -- the single source of truth for the schema and generated client, used by both `apps/web` and `apps/api`.

## Layout

| Path | Purpose |
|:-----|:--------|
| `prisma/schema.prisma` | The schema itself -- single source of truth, 30 models |
| `prisma.config.ts` | Prisma config, lives at the package root (not inside `prisma/`); loads env vars from `apps/web/.env.local` via dotenv |
| `src/client.ts` | Neon serverless adapter singleton (`PrismaNeon`) |
| `src/generated/prisma/` | Generated client output (gitignored, regenerated by `postinstall` / `db:generate`) |
| `src/index.ts` | Package entry point -- re-exports the client singleton and generated types |

**Never import from `@prisma/client` directly** -- always import from `@nextcalc/database`. The generator is `prisma-client` (not the old `prisma-client-js`), and re-exports also pull in `./generated/prisma/models` so TypeScript declaration emit works correctly in consuming packages.

## Prisma CLI

Always run Prisma commands from this package (not the repo root):

```bash
pnpm --filter @nextcalc/database db:generate        # Generate the Prisma client
pnpm --filter @nextcalc/database db:push            # Push schema to the database (dev)
pnpm --filter @nextcalc/database db:migrate         # Create + apply a migration (dev)
pnpm --filter @nextcalc/database db:migrate:deploy  # Apply migrations (production)
pnpm --filter @nextcalc/database db:studio          # Open Prisma Studio (port 5555)
```

## Adapter Gotcha

`PrismaNeon` takes a **config object**, not a `Pool` instance -- Prisma 7's adapter creates its own connection pool internally via `connect()`:

```typescript
// Correct
const adapter = new PrismaNeon({ connectionString: process.env.DATABASE_URL });

// Wrong -- "No database host or connection string" error
const adapter = new PrismaNeon(new Pool({ connectionString: process.env.DATABASE_URL }));
```
